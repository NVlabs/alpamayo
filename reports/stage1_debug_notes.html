Investigation of mlx_vlm cache handling (2026-05-16)

The GPU Address Fault was traced to a race between the caller’s mx.eval(generated) and the internal RoPE/KV cache state managed inside mlx_vlm/generate.py:generate_step.

First-principles triage of GPU Hang (00000003:kIOGPUCommandBufferCallbackErrorHang)

Symptom: Hang occurs on the very first autoregressive step after a 32 k-token vision prefill.

Root cause: A single model.language_model(...) call with 32 k context generates a Metal command buffer containing thousands of kernel dispatches that exceeds the driver’s internal timeout.

Architectural mismatch: mlx_vlm.generate was designed for typical VLM prompts (hundreds to low thousands of tokens). Alpamayo’s CoC prompts routinely exceed 32 k tokens with heavy vision augmentation.

End of archived historical notes. Current implementation uses the manual loop in mlx_port/inference.py.

Failure Cases & Debugging Notes (Moved from main progress report)

1. Analytical combined motion (expected): traj_to_action recovers mixed (accel + kappa) profile with mean error ~0.07–0.19. Documents current numerical limitation of heading-projection + regularized solver.
2. End-to-end inference (new bug): traj_to_action returns Python list from solve_xs_eq_y when called with real dataset shapes inside tokenizer. Blocks history tokenization. CoC generation path still functional.
3. mlx_vlm Qwen3-VL processor nested-list bug (2026-05-18, detailed debug trace): When forcing the strict NVIDIA-style single call processor.apply_chat_template(messages, tokenize=True, return_dict=True), the mlx_vlm processor receives a nested list [[pil0, …, pil15]] instead of a flat list of 16 images. This causes Qwen3VLImageProcessor.__call__ to produce a (16, 1080, 1920, 3) array that reaches _process_one, which expects a 3-D image and raises ValueError: too many values to unpack (expected 3). Debug prints were temporarily inserted in mlx_vlm/models/qwen3_vl/processing_qwen3_vl.py (in Qwen3VLProcessor.__call__ and Qwen3VLImageProcessor.__call__) to capture the exact data flow. The prints showed:
[MLX_VLM_DEBUG] Qwen3VLProcessor.__call__ passing to image_processor: type=list, len=1
[MLX_VLM_DEBUG] Qwen3VLImageProcessor.__call__ received: type=list, len=1; First element type: list
[MLX_VLM_DEBUG] After conversion: 1 items, shapes=[(16,1080,1920,3)]
[MLX_VLM_DEBUG] Calling _process_one on shape (16,1080,1920,3)
This explains why the fallback split path (tokenize=False + explicit images) silently produced the incorrect image_grid_thw with every row [1,68,120] (T=1). The root cause is an assumption in the mlx_vlm port that the images list passed from apply_chat_template is always flat. These debug prints (currently left in the installed mlx_vlm package) can be re-enabled for future diagnosis.

VLM Generation: Manual Loop (Clean Environment) – Historical Record

Status (2026-05-16): Reverted to pristine mlx-vlm==0.5.0 (no patches). All previous patch work on mlx_vlm/generate.py has been discarded.

Current Implementation: Full manual VLM generation loop in inference.py (_run_single_vlm_generation).
• One prefill call with full prompt + vision features.
• Decode loop applies ExpertLogitsProcessor and StopAfterEOS on every step.
• mx.eval(outputs.logits) immediately after prefill (forces KV cache materialization).
• mx.eval(next_token) on the first decode token only (prevents lazy-graph hang).
• Cache object reused on every continuation call (no KV recomputation).

Why manual loop: The high-level mlx_vlm.generate API could not be made safe for Alpamayo’s 32 k-token vision-augmented CoC prompts without repeated Metal timeouts or page faults. The manual loop gives explicit control over synchronization points while preserving exact NVIDIA parity for processors and stopping criteria.
